# Test Spec

测试 `docs/global-spec/` 下的业务功能规格节点，从叶子节点向上递归执行验收标准审查、测试用例匹配/生成、测试执行、状态更新和父节点传播。

## 执行流程

### 步骤 0：参数解析与节点定位

1. 检查用户是否传入了参数
2. **无参数时**：向用户询问：
   - 是否测试整个项目所有业务功能节点
   - 或指定某个节点（提示输入路径或中文标题）
3. **有参数时**，自动判断参数类型：
   - 尝试作为相对路径解析（相对于 `docs/global-spec/`）：
     - 补全 `.md` 后缀
     - 如果不含 `business-nodes/` 或 `tech-nodes/` 前缀，在两个目录下查找
     - 路径存在 → 定位成功
   - 路径解析失败时，作为中文标题搜索：
     - Glob 扫描 `docs/global-spec/**/*.md`（排除 `.bak` 文件）
     - 读取每个文件的 H1 标题行
     - 标题完全匹配或包含参数文本 → 匹配成功
     - 多个匹配 → 列出所有匹配项让用户选择
   - 都无法定位 → 报错并列出所有可用节点
4. **节点类型判断**：
   - 文件名为 `index.md` → 父节点（有子功能，需先递归测子节点）
   - 其他文件名 → 叶子节点（直接测试）
5. **节点适用范围**：
   - 路径以 `business-nodes/` 开头 → 执行完整测试流程
   - 路径以 `tech-nodes/` 开头 → 提示用户转测关联的业务节点，列出从 `**关联技术模块**:` 反向查找的业务节点路径

### 步骤 1：技术栈探测

读取项目根目录的构建文件，推断技术栈信息。探测结果在整个 skill 执行过程中复用。

按以下顺序检测：

1. `pom.xml` 存在 → Java + Maven
   - 测试框架：从 `pom.xml` 的 `<dependencies>` 中查找 JUnit 版本（JUnit 4 或 JUnit 5）
   - 测试目录：`src/test/java/`
   - 单方法执行：`mvn test -Dtest=ClassName#methodName`
   - 单类执行：`mvn test -Dtest=ClassName`
   - 测试文件推算：`src/main/java/**/{ClassName}.java` → `src/test/java/**/{ClassName}Test.java`
   - 测试方法识别：`@Test`、`@ParameterizedTest` 注解的方法，`@DisplayName` 值
2. `build.gradle` 或 `build.gradle.kts` 存在 → Java + Gradle
   - 单方法执行：`gradle test --tests ClassName.methodName`
   - 其他同 Maven
3. `package.json` 存在 → TypeScript/JavaScript
   - 检查 `devDependencies` 中 jest / vitest / mocha
   - 默认使用 jest
   - 测试目录：`__tests__/` 或 `*.test.ts` / `*.spec.ts`
   - 单方法执行：`npx jest -t "test name pattern"`
   - 单文件执行：`npx jest path/to/test.file.ts`
4. `pyproject.toml` 或 `setup.py` 存在 → Python
   - 默认使用 pytest
   - 测试目录：`tests/`
   - 单方法执行：`pytest path/to/test.py::test_func`
   - 单文件执行：`pytest path/to/test.py`
5. 以上都不存在 → 报错"无法识别项目技术栈"

### 步骤 2：递归测试节点（深度优先）

对目标业务节点执行以下完整流程。如果是父节点，先对每个子节点递归执行，全部完成后再执行当前节点自身。

#### 2.0 检查子节点

1. 读取节点文件内容
2. 查找 `## 子功能` 或 `## 功能点` 段落
3. 提取子节点链接列表，格式为 `[中文名](english-name.md)`
4. **按链接出现顺序**，对每个子节点：
   - 构建子节点文件路径（当前节点目录 + 链接文件名）
   - 递归执行步骤 2.0→2.6
5. 所有子节点完成后，继续当前节点的 2.1

#### 2.1 审查验收标准

1. 从节点文件中提取以下段落内容：
   - `**目标**:`
   - `**范围**:` （包括/不包括列表）
   - `**设计**:`
   - `**详细设计**:` （如有）
   - `**设计示例**:` （如有）
   - `**规格要点**:` （如有）
2. 读取 `**验收标准**:` 段落下的所有条目，每条格式为：
   ```
   - [ ] {名称} — {状态}
     - Given: {前置条件}
     - When: {操作}
     - Then: {预期结果}
   ```
3. **完整性检查**：
   - 遍历目标段落，检查每个目标是否有对应的验收标准
   - 遍历范围的"包括"列表，检查每条是否有正向验收项覆盖
   - 遍历范围的"不包括"列表，检查每条是否有反向验收项
   - 检查边界条件覆盖：null/空值、越界、类型不匹配、极端输入
   - 收集所有缺失项为"建议新增验收标准"列表
4. **一致性检查**：
   - 逐条验收标准与目标/范围/设计内容比对
   - 验收标准描述与业务功能描述有出入的标记为"建议修改"
   - 验收标准过时或与当前设计矛盾的标记为"建议删除"
5. 如有任何建议（新增/修改/删除）：
   - 向用户展示完整变更清单，格式：
     ```
     验收标准变更建议:
     + [新增] {名称}
       Given: ...
       When: ...
       Then: ...
     ~ [修改] {名称}: {变更说明}
     - [删除] {名称}: {删除原因}
     ```
   - 等待用户确认
   - 用户确认后，就地修改节点文件的验收标准段落
   - 用户拒绝的项不修改

#### 2.2 审查测试用例

1. 读取 `**测试用例**:` 段落（如有）。格式为表格：
   ```
   | 名称 | 来源 | 测试代码引用 | 测试代码 | 状态 |
   |------|------|-------------|---------|------|
   ```
   如果该段落不存在，视为空表格，后续自动创建。

2. **覆盖度检查**：
   - 遍历 2.1 确认后的验收标准列表
   - 检查每条验收标准在测试用例表格中是否至少有一个对应行（通过"来源"列匹配验收标准名称）
   - 检查边界条件是否有对应测试用例
   - 收集缺失项为"建议新增测试用例"列表

3. **一致性检查**：
   - 逐条测试用例与对应验收标准的 Given-When-Then 比对
   - 测试用例描述与验收标准不一致的标记为"建议修改"或"建议删除"

4. **代码关联**（混合方式）：

   对每条测试用例：

   a. 检查"测试代码引用"列是否已有值（如 `TemplateEngineTest.testBasicRendering`）
      - 已有 → 跳过，使用已有引用

   b. 没有引用时，执行语义匹配：
      - 从节点的 `**关联技术模块**:` 提取链接文本，推算源码类名
      - 根据步骤 1 的技术栈推算测试类路径
      - 读取测试类文件，提取所有测试方法（方法名、DisplayName、断言描述）
      - 将测试用例的名称 + 验收标准的 Given-When-Then 关键词与测试方法信息做语义匹配
      - 关键词重叠度 ≥ 60% → 匹配成功，写入引用到文档

   c. 找不到匹配 → 测试用例的"测试代码引用"标记为"无"，"状态"标记为"未实现"

5. 如有任何建议（新增/修改/删除测试用例）：
   - 向用户展示完整变更清单，格式：
     ```
     测试用例变更建议:
     + [新增] {名称} | 来源: {验收标准名} | 测试代码引用: {匹配结果或"无"}
     ~ [修改] {名称}: {变更说明}
     - [删除] {名称}: {删除原因}
     ```
   - 等待用户确认
   - 用户确认后，就地修改节点文件的测试用例段落
   - 如果测试用例段落不存在，在 `**状态**:` 之前创建完整的测试用例表格

#### 2.3 标记测试状态

就地修改节点文件，将所有状态标记为"测试中"：

1. 将 `**状态**:` 行的值改为 `测试中`
2. 将 `**验收标准**:` 下每条验收标准的 `— {状态}` 改为 `— 测试中`
3. 将 `**测试用例**:` 表格中每行的"状态"列改为 `测试中`

使用 Edit 工具逐个替换，确保格式不变。

#### 2.4 执行测试

**对"未实现"的测试用例生成测试代码：**

1. 收集所有"状态"为"未实现"的测试用例
2. 对每个测试用例，根据其来源验收标准的 Given-When-Then 描述生成测试代码：
   - **Java + JUnit 5**:
     ```java
     @Test
     @DisplayName("{测试用例名称}")
     void test_{methodSuffix}() {
         // Given
         {根据 Given 描述设置前置条件}
         // When
         {根据 When 描述执行操作}
         // Then
         {根据 Then 描述编写断言}
     }
     ```
   - **TypeScript + Jest**:
     ```typescript
     test('{测试用例名称}', () => {
         // Given
         {根据 Given 描述设置前置条件}
         // When
         {根据 When 描述执行操作}
         // Then
         {根据 Then 描述编写断言}
     });
     ```
   - **Python + pytest**:
     ```python
     def test_{method_suffix}():
         # Given
         {根据 Given 描述设置前置条件}
         # When
         {根据 When 描述执行操作}
         # Then
         {根据 Then 描述编写断言}
     ```
3. 确定目标测试文件：
   - 优先写入节点关联技术模块对应的已有测试类
   - 测试类不存在 → 创建新测试类文件
4. 向用户展示所有生成的测试代码，格式：
   ```
   即将生成以下测试代码:

   文件: {测试文件路径}
   ```java
   {测试代码}
   ```
   ```
5. 等待用户确认后写入文件
6. 更新节点文件中对应测试用例的"测试代码引用"和"测试代码"列

**批量执行测试：**

1. 收集所有有测试代码引用的测试用例，按测试类分组
2. 按测试类批量执行：
   - Java: `mvn test -Dtest=ClassName`（超时 60 秒/类）
   - TypeScript: `npx jest TestFile`（超时 60 秒）
   - Python: `pytest path/to/test.py`（超时 60 秒）
3. 解析执行输出，提取每个测试方法的通过/失败状态：
   - Java: 解析 Surefire 输出或 `Tests run: N, Failures: M` 行
   - TypeScript: 解析 jest 输出的 PASS/FAIL 和具体测试方法结果
   - Python: 解析 pytest 输出的 PASSED/FAILED 行
4. 编译错误单独记录：标记为"未通过"，附带编译错误信息
5. 执行超时：标记为"未通过"，附带超时信息

#### 2.5 更新状态

就地修改节点文件，根据测试执行结果更新状态：

**测试用例状态更新规则**：

| 条件 | 状态 |
|------|------|
| 有测试代码且执行通过 | 已通过 |
| 有测试代码但执行未通过 | 未通过 |
| 有测试代码但正在执行 | 测试中 |
| 有测试代码但未执行 | 未测试 |
| 无测试代码 | 未实现 |

**验收标准状态更新规则**（从其关联的测试用例推导）：

| 条件 | 状态 |
|------|------|
| 所有对应测试用例已通过 | 已通过测试 |
| 有测试用例未通过 | 测试未通过 |
| 测试用例正在执行 | 测试中 |
| 有测试用例但未执行 | 未测试 |
| 无对应测试用例 | 未实现 |

**节点状态更新规则**（从验收标准推导）：

| 条件 | 状态 |
|------|------|
| 所有验收标准已通过测试 | 已实现 |
| 有验收标准未通过 | 未测试 |
| 验收标准正在测试 | 测试中 |
| 有验收标准但未测试 | 未测试 |
| 无验收标准 | 未设计 |

使用 Edit 工具逐个替换状态字段，确保格式与替换前一致。

#### 2.6 向上传播状态

1. 从当前节点的文件路径推导父节点路径：
   - 叶子节点 `{dir}/{file}.md` → 父节点 `{dir}/index.md`
   - L3 节点 `business-nodes/{dir}/index.md` → 无业务父节点，直接到 `project.md`
2. 读取父节点文件
3. 在父节点的 `**验收标准**:` 中找到对应当前子节点的验收标准条目：
   - 通过验收标准名称与子节点的 H1 标题或文件名匹配
4. 更新该验收标准的状态为子节点的最终推导状态
5. 重新检查父节点所有验收标准状态，重新推导父节点状态
6. 更新父节点的 `**状态**:` 字段
7. 如果父节点还有父节点（且父节点不是 `business-nodes/` 的直接子目录 `index.md`）→ 递归向上
8. 到达 `project.md` 时：
   - 读取 `project.md` 中的统计表
   - 重新计算各状态数量（遍历所有子节点的当前状态）
   - 更新统计表中的数字

### 步骤 3：完成报告

测试完成后向用户报告：

```
测试完成报告:
- 测试节点: {节点路径}
- 验收标准: {n} 条通过, {n} 条未通过, {n} 条新增
- 测试用例: {n} 个通过, {n} 个未通过, {n} 个新增生成
- 测试执行: {n} 个测试类, 耗时 {n} 秒
- 状态传播: 已更新到 {父节点路径}
- 新生成测试代码: {列出新增的测试方法及其文件路径}
```

如果是全量测试（所有节点），额外报告总汇总。

### 边界情况

| 场景 | 处理方式 |
|------|----------|
| 节点文件不存在 | 报错，Glob 扫描所有 `business-nodes/**/*.md` 并列出可用节点 |
| 节点没有验收标准段落 | 跳过 2.1，直接进入 2.2 基于目标/范围/设计自动生成验收标准 |
| 节点没有测试用例段落 | 在 2.2 中自动创建测试用例表格段落 |
| 验收标准/测试用例无变更 | 跳过确认步骤，直接进入下一步 |
| 测试类不存在 | 在 2.4 中创建新测试类文件，包含 package 声明和 import |
| 测试执行超时 | 标记为"未通过"，报告超时信息 |
| 编译错误 | 标记为"未通过"，报告编译错误详情 |
| 用户指定技术节点 | 提示关联的业务节点路径，建议用户转测业务节点 |
| 多个中文标题匹配 | 列出所有匹配项（文件路径 + 标题），供用户选择 |
| 子节点测试失败 | 继续测试剩余子节点，不中断。父节点状态根据所有子节点综合推导 |
| 用户拒绝所有变更建议 | 按现有验收标准和测试用例继续执行 |
