# Merge Specs

合并项目所有规格信息，生成涵盖项目全局的功能规格文档。

## 执行流程

首先进行模式判断，决定走增量模式还是全量模式：

### 步骤 0：模式判断

1. 检测 `docs/global-spec/project.md` 是否存在
   - 不存在 → **全量模式**
   - 存在 → 继续
2. 检查用户参数
   - 用户传入 `--full` → **全量模式**
   - 无参数 → **增量模式**

- `/sync-specs` → 自动判断：有文档走增量，无文档走全量
- `/sync-specs --full` → 强制全量

### 全量模式流程

严格按以下三阶段顺序执行，每个阶段完成后进入下一阶段。

### 阶段 A：代码扫描（3 个并行子代理）【全量模式】

使用 Agent 工具（subagent_type 设为 `Explore`）**同时启动 3 个并行子代理**，各自独立完成分析。等待全部完成后合并结果。

#### 子代理 A1：项目概览与模块识别

**任务**：读取项目配置文件和源码目录结构，返回项目概要和模块列表。

具体步骤：
1. 读取 `pom.xml`（或 `build.gradle`）、`CLAUDE.md`、`README.md`，提取：
   - 项目名称和一句话描述
   - 编程语言和版本要求
   - 框架和依赖（列出主要依赖，忽略测试/构建工具）
   - 项目入口点
2. 用 Glob 扫描 `src/main/java/` 下的包结构，识别所有模块。对每个模块记录：
   - 模块名称（从包名提取，如 `parser`、`compiler`）
   - 模块包含的类列表（仅类名）
   - 模块的职责描述（从包结构、类名、Javadoc 推断）

以如下格式返回：

```
## 项目概要
- 名称: ...
- 描述: ...
- 语言: ...
- 依赖: ...
- 入口点: ...

## 模块列表

### 模块: {模块名}
- 包路径: ...
- 类列表: {类名1}, {类名2}, ...
- 描述: ...

### 模块: {模块名}
...
```

#### 子代理 A2：源码功能点提取

**任务**：扫描所有 Java 源码文件，提取技术功能点。不分析测试文件，不归纳业务功能。

具体步骤：
1. 用 Glob 扫描 `src/main/java/` 下的所有 `.java` 文件
2. 按模块（包）分组，对每个模块中的每个关键类识别功能点
3. 对每个功能点记录：
   - 功能点名称（简洁的业务描述，如"词法分析"、"类型转换"）
   - 对应的源码类名和方法签名
   - 功能描述（从类注释、Javadoc、方法名推断，1-3 句话）
   - 规格要点（从方法签名、参数类型、返回类型推断关键行为，3-8 条）

以如下格式返回：

```
## 模块: {模块名}
- 包路径: ...

### 功能点: {功能点名}
- 源码: {类名.方法名}
- 描述: ...
- 规格要点:
  1. ...
  2. ...

### 功能点: {功能点名}
...

## 模块: {模块名}
...
```

#### 子代理 A3：测试检测

**任务**：扫描测试目录并执行测试，返回测试状态映射和测试方法清单。

具体步骤：
1. 用 Glob 扫描 `src/test/` 下的所有测试文件，记录测试类与被测类的对应关系
2. 对每个测试类，读取源码提取每个测试方法的名称和注解描述（如 `@Test` 方法的名称、`@DisplayName` 值）
3. 运行 `mvn test`（超时 120 秒），解析输出判断测试是否通过
4. 对每个被测类记录：
   - 被测类名
   - 是否有对应测试类
   - 测试是否通过（有测试且通过 / 有测试但未通过 / 有测试但无法确定 / 无测试类）
   - 测试方法清单（方法名 + 描述）

以如下格式返回：

```
## 测试状态

### {被测类名}
- 测试类: {测试类名 或 "无"}
- 状态: {有测试且通过 / 有测试但未通过 / 有测试但无法确定 / 无测试类}
- 测试方法:
  - {testMethodName}: {DisplayName 或方法名推断的场景描述}
  - {testMethodName}: ...

### {被测类名}
...
```

如果 `mvn test` 超过 120 秒未完成，报告"测试执行超时"，但仍列出检测到的测试类。

#### A1/A2/A3 结果合并

3 个子代理全部返回后，主代理按以下方式合并：

1. 以 A1 的模块列表为基础框架
2. 将 A2 的功能点详情填入对应模块
3. 用 A3 的测试状态映射补充每个功能点的测试覆盖信息（通过功能点的源码类名与 A3 的被测类名做匹配）

合并后得到完整的**技术功能点清单**（含项目概要、模块、功能点描述、规格要点、测试状态），进入阶段 B。

### 阶段 B：文档扫描（按模块分组子代理）【全量模式】

主代理先获取文档列表并建立模块映射，再按模块分组启动子代理扫描。

#### 步骤 1：获取文档列表

1. 使用 Glob 扫描 `docs/superpowers/specs/*.md` 获取所有设计文档路径
2. 使用 Glob 扫描 `docs/superpowers/plans/*.md` 获取所有计划文档路径

#### 步骤 2：建立文档→模块映射

主代理快速扫描每个文档（仅读标题和首段），将文档关联到阶段 A 识别的模块：
- 通过文件名中的关键词匹配模块名（如 `type-cast` 匹配 runtime 模块）
- 通过文档标题中的模块/类名引用匹配

将文档分组为若干模块组，每组包含 1-3 个文档。未匹配到任何模块的文档归入"通用"组。

#### 步骤 3：启动子代理

对每个模块组（含至少一个文档），启动一个 Explore 子代理，传入该组的文档路径列表。

每个子代理逐个读取组内文档，提取：
- 文档标题（第一行 H1）
- 文档类型：design 或 plan
- 关联的功能模块（从文件名关键词和文档内容推断）
- 关联的业务功能点（从文档内容中的用户视角描述关键词推断）
- 目标（从 spec 文档的"目标"/"背景"段落提取）
- 范围（从 spec 文档中显式的范围描述提取）
- 设计决策（从 spec 文档的"设计决策"/"方案选择"段落提取）
- 验收标准（从 spec 文档中"验收标准"/"测试计划"/"AC"段落提取，转化为 Given-When-Then 格式）
- 文档摘要（前几段的核心描述）

返回格式：

```
| 文档路径 | 类型 | 关联模块 | 关联功能点 | 关联业务功能 | 目标 | 范围 | 设计决策 | 验收标准 | 摘要 |
|----------|------|----------|-----------|-------------|------|------|---------|---------|------|
| ... | ... | ... | ... | ... | ... | ... | ... | ... | ... |
```

#### 步骤 4：合并结果

所有子代理返回后，主代理将各组的提取表合并为统一的**文档映射表**。

**合并优先级**：将阶段 B 提取的文档内容与阶段 A 的技术功能点清单合并时，遵循以下规则：
- **交集（文档+代码都有）**：合并两者信息，文档内容放在前面作为权威来源，代码推断作为补充
- **仅有文档**：使用文档内容，业务功能点状态标记为对应的无代码状态（已设计/已计划/未设计）
- **仅有代码**：使用阶段 A 的技术功能点数据

### 阶段 C：合并输出（分步生成）【全量模式】

将阶段 A 和阶段 B 的结果合并，分步生成多文件树形结构的全局规格文档。

**输出目录**：`docs/global-spec/`，包含 `project.md`、`business-nodes/` 和 `tech-nodes/` 子目录。

**执行步骤**：
1. **C1 构建节点索引**：归纳业务功能树和技术模块树，构建轻量节点索引表
2. **C2 生成 project.md**：主代理直接生成项目根文件
3. **C3 逐文件生成节点**：对每个节点启动子代理，一次只生成一个文件

#### 业务功能归纳

主代理基于阶段 A 的技术功能点清单和阶段 B 的文档映射表，从用户/业务视角归纳出高层业务功能。对每个业务功能：

1. 确定业务功能名称（从用户能理解的概念出发，不使用技术术语）
2. 确定功能描述（从用户角度描述这个功能解决什么问题）
3. 确定子功能列表（层级可递归拆分，每个子功能关联到具体的技术功能点，跨模块）
4. 归纳规格要点（从用户角度归纳的关键行为，3-5 条）

注意：一个业务子功能可以关联多个技术功能点（跨模块），一个技术功能点也可能被多个业务子功能引用。

#### C1：构建节点索引

主代理基于阶段 A 的技术功能点清单和阶段 B 的文档映射表，构建轻量级**节点索引表**。此步骤由主代理直接完成，不需要子代理。

节点索引表仅包含元数据，不含详细内容：

```
| 节点 ID | 类型 | 中文标题 | 英文名 | 父节点 ID | 层级 | 关联技术功能点 IDs | 状态 | 关联文档路径 |
```

同时确定：
- 业务功能树结构（L3/L4/L5+ 层级关系）
- 技术模块树结构
- 中文标题 → 英文命名的映射（kebab-case，简洁英文翻译）

**业务功能点推断规则**：

对每个业务功能点（含所有层级子功能），目标/范围/设计/详细设计/设计示例/验收标准的推断遵循以下规则：

1. **目标**：从功能描述和方法签名推断该功能的业务目的，1-3 句话
2. **范围**：从关联的技术功能点推断涵盖范围，排除相邻功能的边界。包括 2-5 条，不包括 1-3 条
3. **设计**：从代码实现模式（如类层次、设计模式、API 设计）推断关键设计决策，格式为 `{决策点}: {选择} — {理由}`
4. **详细设计**：描述该功能点所有设计完成后的最终形态。从用户视角描绘"完成后它是什么样的"。例如：
   - 对新语法特性：描述该语法的最终语法形式、支持的变体、完整的表达式写法
   - 对 API 功能：描述最终的 API 调用方式、参数列表、返回值形式
   - 对配置能力：描述配置项的完整结构、可用选项、默认值
   - 如果功能尚未实现，从设计和目标推断最终形态；如果已实现，从代码和测试用例中还原最终形态
5. **设计示例**：提供该功能点在实现后的典型使用示例。从测试代码中提取真实的调用示例，或从文档中提取用法示例。示例应覆盖：
   - 基本用法（最常见的场景）
   - 进阶用法（参数组合、边界值等）
   - 如果功能尚未实现，从设计推断合理的使用示例
   - 每个示例包含代码片段和预期输出/效果说明
6. **验收标准**：仅从该业务节点的规格内容（描述、目标、范围、设计、详细设计）推断，不从测试代码反推。推导方向严格单向：规格 → 验收标准 → 测试用例。推断时：
   - 从目标推断核心验收项："这个功能要做到什么程度才算完成"
   - 从范围推断边界验收项：每条"包括"至少对应一个正向验收项，每条"不包括"至少对应一个反向验收项
   - 从设计和详细设计推断实现细节验收项：关键设计决策和最终形态的具体行为验证
   - 覆盖所有边界条件：空值/null、越界、类型不匹配、极端输入、并发等
   - 每条验收标准以 Given-When-Then 格式描述
   - 验收标准的状态推断（基于其对应的测试用例状态）：

| 条件 | 状态 |
|------|------|
| 所有对应测试用例都已通过 | 已通过测试 |
| 有测试用例但至少一个未通过 | 测试未通过 |
| 有测试用例但状态为测试中或未测试 | 未测试 |
| 无对应测试用例 | 未实现 |

7. **测试用例**：仅从验收标准推导生成，每条验收标准至少对应一个测试用例。对每个测试用例记录：
   - **名称**：简洁描述测试场景，如"空字符串输入应返回空结果"
   - **来源**：标注来自哪条验收标准（用验收标准名称引用）
   - **测试代码引用**：如果项目中已有对应测试代码，标注文件路径和方法名（如 `{TestClass}.{testMethod}`）；如无则标注"无"
   - **测试代码**：如果项目中没有已实现的测试代码，生成可执行的测试用例代码；如果已有实现的测试代码，此项可省略
   - **状态**：未实现 / 未测试 / 测试中 / 未通过 / 已通过
   - 状态推断规则：

| 条件 | 状态 |
|------|------|
| 有测试代码且执行通过 | 已通过 |
| 有测试代码但执行未通过 | 未通过 |
| 有测试代码但正在调试/修复中 | 测试中 |
| 有测试代码但未执行 | 未测试 |
| 无测试代码 | 未实现 |

   - **测试代码引用匹配**：A3 子代理扫描测试类时，同时收集每个测试方法名和其对应的测试场景描述。在生成测试用例时，通过测试方法名中的关键词与验收标准描述进行匹配，建立验收标准 → 已有测试方法的引用关系。

每个子功能同样需要完整的目标/范围/设计/详细设计/设计示例/验收标准/测试用例输出，粒度随层级递进细化。

#### 状态推断

对每个功能点，按以下优先级推断状态：

| 优先级 | 条件 | 状态 |
|--------|------|------|
| 1 | 有代码 + 有测试 + 测试通过 | 已实现 |
| 2 | 有代码 + 有测试 + 测试未通过 | 未测试 |
| 3 | 有代码 + 无测试 | 未测试 |
| 4 | 无代码 + 有 spec + 有 plan | 已计划 |
| 5 | 无代码 + 有 spec + 无 plan | 已设计 |
| 6 | 仅在全局文档中有需求描述，无代码无 spec 无 plan | 未设计 |

业务功能点的状态由其关联的技术功能点状态推导，取最保守（优先级最低）的状态：

| 优先级（低→高） | 状态 |
|------------------|------|
| 1 | 未设计 |
| 2 | 已设计 |
| 3 | 已计划 |
| 4 | 未测试 |
| 5 | 已实现 |

规则：业务功能点的状态 = 其所有关联技术功能点中优先级最低的那个。例如关联了"已实现"和"未测试"两个技术功能点，则业务功能点状态为"未测试"。如果业务子功能还有更深的子功能，则递归取最保守状态。

注意：有代码但无 spec/plan 的功能点，仍从代码生成规格描述，状态根据测试情况标记。

#### 验收标准与测试用例状态体系

验收标准和测试用例采用独立的状态体系，与功能点状态互不影响：

**功能点状态**（现有，不变）：已实现 / 未测试 / 已设计 / 已计划 / 未设计

**验收标准状态**（基于其对应测试用例的状态推导）：未实现 / 未测试 / 测试未通过 / 已通过测试

**测试用例状态**（独立推断）：未实现 / 未测试 / 测试中 / 未通过 / 已通过

**推导链路**：规格内容 → 验收标准 → 测试用例（严格单向，不可反向）

#### 父子联动规则

- 子节点状态独立：每个节点（含子节点）的验收标准状态独立推断，不受父节点影响
- 父节点受子节点影响：父节点的"完成"需要自身所有验收标准都已通过测试 **且** 所有后代节点的完成条件都满足
- 输出格式中每个节点如实显示自身的验收标准状态，联动规则体现在完成后报告中

#### 文档匹配

将阶段 B 的文档匹配到阶段 A 的功能点：
- 通过文件名关键词匹配（如 `error-message-format` 匹配"错误消息格式"功能点）
- 通过文档内容中的模块/类名引用匹配
- 通过文档内容中的用户视角描述关键词匹配业务功能点
- 一个文档可关联多个功能点，一个功能点可关联多个文档

#### C2：生成 project.md

主代理直接生成 `project.md`，只需项目概要、状态统计、业务功能索引和技术模块索引。确保 `docs/global-spec/` 目录存在后写入。

#### C3：逐文件生成节点

对节点索引表中的每个节点，按以下顺序逐个生成：

1. 所有 L3 业务节点（需要知道子节点列表来写索引）
2. 所有 L4+ 业务节点
3. 所有 L3 技术模块节点
4. 所有 L4+ 技术功能点节点

对每个节点：

1. 主代理从索引表和阶段 A+B 原始结果中提取该节点的关联数据
2. 启动一个 `general-purpose` 子代理，传入：
   - 节点的关联数据（描述、功能点、测试状态、文档摘要）
   - 目标文件路径（基于节点英文名和父节点路径推导）
   - 文件格式模板（参见下方"输出文件"部分）
   - 如果是 L3 节点，附带子节点英文名和描述列表（用于生成"子功能"索引）
   - 如果有旧文件迁移的手动内容，一并传入
3. 子代理生成单个文件并写入目标路径
4. 主代理确认写入成功，继续下一个节点

#### 智能合并（多文件迁移）

**旧文件检测**：如果 `{项目目录}/docs/global-spec/global-spec.md` 已存在：

1. 读取旧文件内容
2. 提取所有 `<!-- manual-start -->` 和 `<!-- manual-end -->` 之间的用户手动内容
   - 手动内容的匹配 key 为"节点路径"，格式为 `{视角}/{L3节点}/{L4节点}/...`，如 `业务视角/模板语法特性/表达式语法` 或 `技术视角/parser/词法分析`
3. 将节点路径映射到新的多文件结构路径（如 `业务视角/模板渲染引擎/基础模板渲染` → `business-nodes/template-rendering/basic-rendering.md`）
4. 在 C3 步骤中，为对应节点传入手动内容，子代理将其还原到新文件的 `<!-- manual-start -->` / `<!-- manual-end -->` 区域
5. 对于代码中已不存在但文档中记录的功能点，在标题后添加 `<!-- removed -->` 标记并保留
6. 对于新增的功能点，正常追加
7. 迁移完成后，将旧的 `global-spec.md` 重命名为 `global-spec.md.bak`（保留备份）

如果旧文件不存在，直接创建新文件结构。

### 增量模式流程

仅当步骤 0 判定为增量模式时执行以下流程。

#### 步骤 1：构建内存索引

主代理直接执行，不启动子代理。

1. Glob 扫描 `docs/global-spec/**/*.md`，排除 `global-spec.md.bak`
2. 逐文件读取，提取元数据：
   - `> sync:` 行 → gitCommit
   - `源码位置` 字段 → 源码文件路径列表
   - 标题行的 `[状态]` 标签 → 当前状态
   - `关联技术模块` 中的链接 → 业务→技术节点关联
3. 从 `project.md` 读取 `> 同步分支:` 行中的 commit 作为 diff 基线
4. 构建内存索引：
   - `技术节点ID → {源码文件路径列表, 测试文件路径列表(推算), 当前状态, gitCommit}`
   - `业务节点ID → {关联技术节点ID列表, 当前状态, gitCommit}`

测试文件推算规则：`src/main/java/**/{ClassName}.java` → `src/test/java/**/{ClassName}Test.java`

#### 步骤 2：变更范围检测

主代理直接执行：

1. `git diff --name-only <project.md中的commit>..HEAD` 获取所有变更文件
2. 如果基线 commit 不在 git 历史中（分支切换等场景），使用 `git merge-base <commit> HEAD` 找到分叉点作为基线
3. 将变更文件分为三类：源码（`src/main/java/`）、测试（`src/test/`）、文档（`docs/superpowers/`）
4. 匹配到内存索引中的节点，生成三个集合：
   - `affectedTechNodes`：源码或测试文件有变动的技术节点
   - `affectedBizNodes`：通过技术节点传导影响的业务节点（遍历 `业务→技术` 映射，业务节点状态 = 关联技术节点中最保守状态）
   - `newDocs`：`docs/superpowers/` 中新增的文档（不在已有节点的关联文档列表中）
5. 三个集合都为空 → 报告"无变更，无需更新"，结束

#### 步骤 3：条件化阶段 A（增量）

**A2 缩小扫描**（仅在 `affectedTechNodes` 非空时执行）：

- 按受影响节点分组为若干模块组
- 对每个模块组启动一个 Explore 子代理
- 子代理 prompt 限定：仅扫描该模块下 git diff 标记的变更文件
- 任务：
  1. 提取变更文件的功能点（格式与全量 A2 一致）
  2. 检测新增的类/方法：扫描受影响模块的所有 `.java` 文件，对比内存索引中该模块已有的 `源码位置` 列表，索引中不存在的类即为新增技术节点
- 新增技术节点的初始数据：功能点名称、描述、规格要点从源码推断，状态默认为"未测试"

**A3 缩小测试**（仅当测试文件有变动时执行）：

- 收集受影响的测试类名
- 逐个执行 `mvn test -Dtest=XxxTest`（可并行）
- 更新对应节点的测试状态

**A1 跳过**：增量模式下不重新扫描项目概览和模块结构。

**新增业务节点检测**（在 A2 完成后由主代理执行）：

1. 检查新增技术节点是否被现有业务节点覆盖
2. 已覆盖 → 仅更新现有业务节点的关联和技术传导状态
3. 未覆盖 → 主代理判断是否需要新建业务节点（独立功能粒度）或合并到现有节点
4. 新建的业务节点标记为待生成，在步骤 5 中走完整 C3 生成流程

#### 步骤 4：条件化阶段 B（增量）

- `newDocs` 为空 → 完全跳过阶段 B
- `newDocs` 非空 → 对新文档启动子代理扫描（同全量阶段 B 的子代理 prompt 格式，但只处理新文件列表）
- 合并新文档的关联信息到内存索引

#### 步骤 5：条件化阶段 C（增量）

范围缩小的阶段 C，仅处理受影响和新增节点。

1. **C1 构建节点索引**：仅重新构建受影响节点的索引条目。新增节点追加到索引，已有节点更新状态
2. **C2 生成 project.md**：主代理直接生成。更新统计数字（可能因状态变化或新增节点而改变）+ 更新 `> 同步分支:` 为当前分支和最新 commit
3. **C3 逐文件生成节点**：仅对以下节点启动子代理重新生成：
   - `affectedTechNodes` 中的技术节点
   - `affectedBizNodes` 中的业务节点
   - 新增的技术节点和业务节点
   - 每个生成的节点文件包含 `> sync: {branch} @ {commit} · {date}` 行
   - 未受影响的节点文件**不读取、不写入**
   - 子代理 prompt 和文件格式模板与全量模式一致（参见下方"输出文件"部分）

### 边界情况处理（增量模式）

| 场景 | 处理方式 |
|------|----------|
| project.md 存在但部分节点文件缺失 | 对缺失节点回退到全量扫描该节点，视为新增 |
| 节点文件的 sync commit 在 git 历史中找不到 | 以 project.md 的 commit 为准，全量扫描该节点 |
| 分支切换后首次增量 | 用 `git merge-base <project.md的commit> HEAD` 找到分叉点作为 diff 基线 |
| 节点文件格式损坏（无法解析 sync 行） | 视为该节点需要全量重新生成 |
| 所有节点 commit 一致但 project.md commit 不同 | 以 project.md 为准更新基线 |
