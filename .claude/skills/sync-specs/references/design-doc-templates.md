# 设计文档模板

根据节点类型（自动检测），选择对应的模板生成设计文档。模板中的章节如无对应代码内容则直接省略不写。

## 节点类型自动检测

通过分析节点的 `syncMeta.sourceFiles` 自动判断：

占比按源文件数量计算（不是代码行数）。扩展名不在识别范围内的文件（如配置文件、文档）不计入分母。

| 类型 | 判断规则 |
|------|---------|
| project | 根节点（level=1） |
| frontend | 前端文件占比 ≥ 80%（`.tsx`/`.jsx`/`.vue`/`.css`/`.scss`/`.less` 等扩展名，或 `components/`/`pages/`/`hooks/` 目录下的文件） |
| backend | 后端文件占比 ≥ 80%（`routes/`/`services/`/`models/`/`controllers/`/`middleware/` 等目录，或 `.go`/`.java`/`.py` 等后端语言文件） |
| fullstack | 前后端文件混合，任一侧不足 80% |

无法判定类型时（如 sourceFiles 仅含配置文件或文档），默认使用 fullstack 类型。

中间节点（有子节点）：生成该业务域的总体设计概述，涵盖子节点涉及的所有技术面。
叶子节点：只生成该功能点的具体设计，范围聚焦不扩散。

## 模板

### project — 项目架构设计

```markdown
# {项目名} 架构设计

## 技术栈
<!-- 列出前端、后端、数据库、关键依赖的名称和版本 -->

## 设计原则
<!-- 项目遵循的核心设计原则（如单一职责、依赖注入等） -->

## 编码规范
<!-- 命名规范、文件组织规范、代码风格等 -->

## 架构图
<!-- 使用 Mermaid 或文字描述系统整体架构 -->

## 项目结构
<!-- 关键目录及其职责说明 -->

## 关键模块
<!-- 各核心模块的职责和接口说明 -->

## 部署架构
<!-- 如有，描述部署方式和环境配置 -->
```

### frontend — 前端设计

```markdown
# {功能名} 前端设计

## 布局
<!-- 页面/组件的整体布局结构 -->

## 路由与导航
<!-- 页面路由结构、导航方式（侧边栏/顶栏/面包屑）、路由守卫 -->

## 组件树
<!-- 组件层级关系和 props/events 接口 -->

## 样式与主题
<!-- 颜色方案、字体、间距、主题变量等 -->

## 交互方式
<!-- 点击、悬停、拖拽等行为；动画/过渡效果；键盘快捷键 -->

## 状态管理
<!-- 使用的 store、状态流转、数据获取方式 -->
```

### backend — 后端设计

```markdown
# {功能名} 后端设计

## API 设计
<!-- 表格形式：方法 | 路径 | 请求体 | 响应体 | 说明 -->

## 数据库表结构
<!-- 表名、字段、索引、关联关系 -->

## 业务流程
<!-- Mermaid 流程图或文字描述数据/控制流 -->

## 中间件/拦截器
<!-- 认证、日志、错误处理等中间件说明 -->

## 认证与授权
<!-- 认证机制（JWT/Session/OAuth）、权限模型（RBAC/ABAC）、鉴权流程 -->

## 错误处理
<!-- 错误码定义、异常处理策略 -->
```

### fullstack — 全栈设计

```markdown
# {功能名} 全栈设计

## 架构概览
<!-- 前后端如何协作：API 契约（请求/响应格式）、数据序列化格式、认证如何在前后端之间传递、跨域策略 -->

## 前端设计
<!-- 同 frontend 模板的章节，按需裁剪 -->

## 后端设计
<!-- 同 backend 模板的章节，按需裁剪 -->
```

全栈节点按实际代码侧重裁剪前后端章节，不强制所有章节都出现。

## 生成原则

1. 从代码中提取事实，不捏造不存在的内容
2. 章节标题固定，但无内容的章节省略不写
3. 代码路径、API 路径等具体信息必须与源文件一致
4. Mermaid 图能用则用，没有足够信息则用文字描述
5. 设计文档长度应与功能复杂度匹配——叶子节点可能只有十几行，根节点可能有上百行
