# 设计文档：sync-specs 树形变更概览与无深度限制

日期：2026-05-13

## 背景

sync-specs skill 存在两个问题：

1. 同步计划的"变更概览"使用表格格式，无法体现节点的层级关系，与后面的"节点树结构"割裂
2. 规格树隐式限制在 3 层以内，导致 `project-management`（包含创建/打开/克隆/删除四个独立操作）和 `smart-tools`（包含 10+ 种独立工具卡片）等功能过粗的节点被错误标记为叶子

## 改动 1：变更概览改为树形格式

### 当前格式

```markdown
## 变更概览

| 节点 | 操作 | 原因 |
|------|------|------|
| {node-id} | {新增/更新/删除/不变} | {原因} |

(末尾还有"新增节点"和"删除节点"两个汇总表格)
```

### 新格式

```markdown
## 变更概览

{节点名} (L{层级}, {操作}){ - 简述}
├── {子节点名} (L{层级}, {操作}){ - 简述}
│   └── ...
└── ...
```

操作标记：新增、更新、删除、不变。
简述可选，用于补充节点含义。

### 删除的冗余部分

"新增节点"和"删除节点"两个汇总表删除。树形已完整展示所有节点和操作类型，汇总表是冗余的。

"详细变更"部分保留不变（每个节点的关联文件、变更原因、预计影响）。

## 改动 2：替换叶子节点终止条件

### 当前终止条件（删除）

```
- 节点描述的功能已足够简单，用户只需一步操作即可完成
- 继续细分后子节点不再有独立的用户价值
- 节点关联的代码已是单文件或单组件级别
```

### 新终止条件

```markdown
**叶子节点判定原则：**

一个节点是叶子节点，当且仅当它描述的功能满足以下全部条件：
1. 它是一个完整的、用户可独立使用的最小功能单位
2. 继续拆分后，子项将不再是完整的功能（即拆开后用户无法独立使用其中任何一个子项）

**原则方向：** 从树根到叶子节点：高层次 → 低层次，抽象 → 具体，模糊 → 明确，大范围 → 小范围。
```

不设最大层数限制。层数完全由业务逻辑的自然拆分决定。

### 示例

**叶子节点（不应继续拆分）：**
- 打开项目 — 完整的最小功能，拆为"选择文件夹"+"确认"没有独立用户价值
- Bash 命令卡片 — 展示命令执行和结果，用户独立感知的最小单位
- 发送消息 — 用户输入文本并发送给 Agent 的完整操作

**非叶子节点（应继续拆分）：**
- 项目管理 → 项目创建、打开项目、克隆项目、删除项目
- 智能工具 → Bash 命令卡片、文件读取卡片、文件编辑卡片、...
- 快捷操作 → 快捷按钮、斜杠命令

## 受影响的 SKILL.md 位置

| 位置 | 改动内容 |
|------|---------|
| 节点识别原则 | 补充"功能节点是否需要继续拆分"的判断逻辑，与叶子判定原则统一 |
| Step 1 递归发现 | 替换递归终止条件为新定义 |
| Step 2 计划模板 | 变更概览从表格改树形，删除"新增节点""删除节点"汇总 |
| Step 2.5 检查项 2 | 微调文字以匹配新的叶子节点定义 |
| Step 5.0 检查项 3 | 更新判断标准为匹配新叶子节点定义 |
| 存储结构部分 | 去掉暗示层数限制的注释 |
| Agent 行为规则表 | 更新计划/计划 Review 行的描述 |

## 不改动

- `references/node-schema.md` 的 JSON Schema（level 字段本身不受限制）
- acceptanceCriteria/testCases 格式
- syncMeta 格式
- treeScenario 枚举值（`single/flat/multi-functional/multi-domain` 定义合理）
- Step 3/4/6 的流程逻辑（这些步骤按节点逐个处理，不关心层数）
