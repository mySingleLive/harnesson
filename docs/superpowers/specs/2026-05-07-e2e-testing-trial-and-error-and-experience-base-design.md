# E2E Testing Skill: 试错循环与经验库设计

## 概述

对 e2e-testing skill 进行两项增强：

1. **统一问题处理流程**：将现有的自愈循环扩展为三层级处理（试错→升级用户→代码修复），操作失败时先换方式尝试，再升级用户，最后才修复代码
2. **经验库 base.md**：将成功的操作流程（包括试错成功和用户指导成功）保存为经验，供后续测试参考；在计划阶段发现多路径时让用户交互选择

## 修改范围

仅修改 `.claude/skills/e2e-testing/SKILL.md`，新增 `docs/e2e-testing/base.md` 经验库文件。

## 一、统一问题处理流程

将现有的"自愈循环"扩展为统一的三层级问题处理机制。不同层级的触发条件不同：

### 流程图

```
操作/验证失败
    │
    ▼
┌─────────────────────────────────────┐
│ 层级1: 试错循环                      │
│ (仅 Given/When 操作步骤失败时触发)    │
│ 换选择器/操作方式/导航路径            │
│ ├─ 成功 → 保存经验到 base.md → 继续  │
│ └─ 3次失败 ↓                        │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 层级2: 升级用户                      │
│ 向用户描述问题(截图+实际状态)         │
│ 请用户提供操作指导                    │
│ ├─ 成功 → 保存经验到 base.md → 继续  │
│ └─ 用户无法解决 ↓                   │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 层级3: 代码修复(原自愈循环)           │
│ 定位根因 → 修复代码 → 重新验证        │
│ ├─ 修复成功 → 继续                  │
│ └─ 3次失败 → 终止测试，报告用户      │
└─────────────────────────────────────┘
```

### 层级1: 试错循环

**触发条件**：Given 或 When 操作步骤执行失败（元素找不到、操作无响应、未进入预期状态等）

**行为**：
- 分析失败原因，换一种操作方式重试
- 可尝试的变化：不同选择器策略（CSS→XPath）、不同操作方式（click→keyboard_press）、不同导航路径、先等待再操作等
- 每次尝试前可截图查看当前页面状态，调整策略
- 成功后立即将成功的操作流程保存到 base.md

**限制**：同一操作最多尝试 3 种不同方式

### 层级2: 升级用户

**触发条件**：试错循环 3 次仍失败

**行为**：
- 向用户展示：问题描述、已尝试的方式（带截图）、当前页面状态
- 请用户提供操作指导："这个操作应该怎么完成？"
- 按用户指导操作，成功后立即保存到 base.md
- 用户无法解决则进入层级3

### 层级3: 代码修复

**触发条件**：
- 层级2 用户无法解决
- **或** Then 验证步骤失败（Then 失败不经过层级1和层级2，直接进层级3）

**行为**：与现有自愈循环完全一致（定位根因→修复代码→重新验证，最多3次）

### 与现有流程的关系

- **现有自愈循环** → 变为层级3，逻辑不变
- **Then 验证失败** → 直接进入层级3（代码修复），不经过试错
- **Given/When 操作失败** → 层级1→层级2→层级3 逐级升级

## 二、经验库 base.md

### 文件位置

`{项目目录}/docs/e2e-testing/base.md`

### 文件格式

```markdown
# E2E 测试经验库

## 导航类

### 进入项目首页
- **策略**：直接导航到应用根 URL
- **关键步骤**：navigate http://localhost:5173 → await_element 等待页面加载
- **注意**：确保开发服务器已启动

### 选择特定项目
- **策略 A（推荐）**：项目列表页 → select 下拉选择
  - 关键步骤：await_element 项目列表 → select 选择目标项目
- **策略 B**：项目列表页 → click 项目名称
  - 关键步骤：await_element 项目列表 → XPath 定位项目名 → click

## 状态进入类

### 触发 Agent 对话
- **策略**：选择项目 → 点击 Agent 聊天入口 → 等待对话界面加载
- **关键步骤**：click 聊天图标 → await_element 输入框 → 验证 ThinkingBar 状态
- **注意**：需要先选择项目才能进入 Agent 对话

## 操作类

### 输入框提交
- **策略**：type 输入内容 → keyboard_press Enter
- **注意**：禁止使用 \n 字符
```

### 记录粒度

粗粒度策略描述 + 中粒度关键步骤：
- **粗粒度**：用一句话描述达成目标的策略（如"项目列表页 → select 下拉选择"）
- **中粒度**：列出关键操作步骤和使用的工具（如"await_element 项目列表 → select 选择目标项目"）
- **不记录**：具体选择器、精确等待时间等易变细节

### 初始化

首次使用时如果 `docs/e2e-testing/base.md` 不存在，创建空文件（仅含标题 `# E2E 测试经验库`）。后续测试中通过试错和用户指导逐步积累经验。

### 读写规则

**读取时机**：
1. 阶段一 Step 2（理解用户意图后、澄清问题前）读取 base.md
2. 阶段二执行每个 Given/When 前，参考 base.md 中同类场景的经验

**写入时机**：
- 层级1 试错成功 → 立即追加
- 层级2 用户指导成功 → 立即追加

**去重规则**：写入前检查是否已有相同场景的记录。如有：
- 已有条目只有一条策略 → 追加新策略（变为策略 B）
- 已有条目有多条策略 → 追加新策略（策略 C/D/...）
- 新策略与已有某条策略本质相同 → 不重复添加

## 三、计划阶段多路径选择

在阶段一的澄清问题步骤中，增加多路径选择机制。

### 触发条件

思考推理中发现某目标（进入某页面、进入某状态、完成某操作）有多种操作路径时。

### 选择流程

1. 读取 base.md，查看是否有该场景的记录
2. 汇总所有可用路径：base.md 中已有的 + 新发现的路径
3. 判断是否需要用户选择：
   - base.md 中记录了该场景且只有一条策略 → 直接采用，不问用户
   - 存在 2 条及以上路径 → 让用户选择
4. 推荐规则：
   - base.md 中的路径标记"(推荐)"
   - 如果 base.md 中同一场景有多条路径，推荐最短路径
5. 用 AskUserQuestion 呈现选项，被选路径写进测试计划文档

### 呈现格式

```
进入 Agent 对话页面有以下路径：
- A: 点击侧边栏聊天图标 → 等待对话界面加载 (推荐, base.md)
- B: 直接导航 /agent/chat URL
```
