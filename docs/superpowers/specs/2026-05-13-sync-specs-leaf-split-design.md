# 设计：增强 sync-specs 叶子节点拆分能力

**日期：** 2026-05-13
**范围：** 仅修改 `SKILL.md` 中「节点识别原则」的叶子节点判定部分
**问题：** 递归节点发现停在过高层级（如 Agent Session 描述为"创建、管理和切换"却未拆分）

## 方案

方案 A：扩展现有叶子判定标准，加入动词枚举检查和代码交叉验证。

## 具体改动

### 1. 动词枚举检查（新增第三条判定条件）

在现有两条叶子判定条件之后，新增：

> 3. 扫描节点的 summary 和 goals，提取其中描述用户操作的动词/动宾短语。若同一节点包含 2+ 个独立用户操作（如"创建、切换、删除"），每个操作必须是用户可独立触发和完成的 → 该节点不是叶子，需按操作拆分。

新增示例表：

| 节点 | summary 中的操作枚举 | 判定 | 拆分结果 |
|------|---------------------|------|----------|
| Agent Session | 创建、管理、切换会话 | 3 个独立操作 → 非叶子 | 创建会话 / 切换会话 / 删除会话 |
| Message Input | 富文本输入、图片上传、键盘快捷键 | 3 个独立操作 → 非叶子 | 文本输入 / 图片上传 / 快捷键操作 |
| 发送消息 | 输入文本并发送给 Agent | 1 个完整操作 → 叶子 | 不拆分 |
| Tool Execution Display | Bash/Read/Write/Edit/Glob/Grep 卡片 | 多个独立工具 → 非叶子 | 每个工具类型一个子节点 |

### 2. 代码交叉验证

对每个候选子节点执行验证：

- 检查 sourceFiles 中是否存在独立 API 路由（如 `POST /agents` vs `DELETE /agents/:id`）
- 检查是否存在独立 UI 组件（如 `CreateSessionModal` vs `SessionSwitcher`）
- 检查是否存在独立 store action（如 `createSession` vs `deleteSession`）

### 3. 未实现节点处理规则

代码验证后两种结果：

- 代码中有对应实现 → 子节点正常创建，status 由后续步骤自动检测
- 代码中无对应实现 → 仍创建子节点，status 设为 `draft`，summary 中标注"功能待实现"

### 4. 补充中粒度拆分示例

在现有非叶子节点示例中补充操作级拆分：

| 节点 | 类型 | 拆分 |
|------|------|------|
| Agent Session | 业务功能（非叶子） | → 创建会话 / 切换会话 / 删除会话（L4 操作级） |
| Tool Execution Display | 业务功能（非叶子） | → Bash 卡片 / 文件读取卡片 / 文件编辑卡片 / ...（L4） |
| Message Display | 业务功能（非叶子） | → 文本消息渲染 / 思考指示器 / 用户问答面板（L4） |

## 不改的部分

- Step 1/2.5/5.0 的流程逻辑不变——它们已引用「节点识别原则」，加强原则后自动生效
- 节点 Schema 不变
- 其他步骤不变
