# E2E 测试计划：AskUserQuestion 键盘导航与视觉增强

## 测试简述

测试 AskUserQuestionPanel 组件的键盘导航功能（↑↓、Ctrl+P/N）、聚焦高亮视觉状态、鼠标悬停反馈、自定义回答输入框自动聚焦、单选点击延迟提交、以及多选模式的键盘操作。通过 brainstorming skill 触发 Agent 产生 AskUserQuestion 交互来验证这些功能。

## 测试目标列表

- [ ] 目标1：触发 AskUserQuestion 面板并验证面板结构正确渲染
- [ ] 目标2：验证键盘 ↓↑ 导航在选项间移动焦点，并显示紫色聚焦高亮（边框 + 外发光）
- [ ] 目标3：验证 Ctrl+N / Ctrl+P 导航功能
- [ ] 目标4：验证 Escape 键重置焦点（无高亮）
- [ ] 目标5：验证鼠标悬停显示柔和高亮，且不覆盖键盘聚焦态
- [ ] 目标6：验证单选模式下 Enter 提交当前聚焦选项
- [ ] 目标7：验证单选模式下点击选项先高亮，300ms 后提交
- [ ] 目标8：验证键盘可导航到自定义回答项，输入框自动获得焦点
- [ ] 目标9：验证自定义回答输入 Enter 提交
- [ ] 目标10：验证多选模式下 Space 切换勾选状态

## 测试方案

从首页开始 → 通过顶部栏下拉菜单选择项目 → 发送 brainstorming 命令触发 Agent → 等待 AskUserQuestion 面板出现 → 使用键盘操作测试导航和交互功能。前置条件：开发服务器已启动（http://localhost:5173）。

### 项目选择操作路径

首页（`/`）即为 NewSessionPage，包含聊天输入框，但输入框在未选择项目时为禁用状态（placeholder 显示"请先选择或创建一个项目"）。选择项目后输入框可用。

**具体步骤：**
1. 导航到 http://localhost:5173 → 页面显示首页，包含 HARNESSON 标题和聊天输入区域
2. 点击顶部栏的项目下拉菜单（Folder 图标 + ChevronDown 图标，显示当前项目名或"No Project"）
3. 在下拉列表中点击"harnesson"项目（列表中每个项目显示项目名和路径）
4. 下拉菜单关闭，项目选中完成，输入框变为可用（placeholder 变为"Message Harnesson... Type / for commands"）

利用 brainstorming skill 的多轮问答特性，可以在一个测试周期内测试多个键盘导航场景。如果第一个问题是单选模式，测试完单选交互后，等待后续问题测试多选模式（若出现）。

## 测试用例列表

### 目标 1：触发 AskUserQuestion 面板

**测试用例 1.1：发送 brainstorming 命令触发 AskUserQuestion**
- **测试数据**：命令 `/superpowers:brainstorming 为聊天界面添加快捷键支持`
- **Given**：导航到 http://localhost:5173 → 等待首页加载（显示 HARNESSON 标题和聊天输入区域）→ 点击顶部栏项目下拉菜单（Folder + ChevronDown 图标）→ 点击"harnesson"项目 → 等待输入框变为可用（placeholder 变为"Message Harnesson..."）
- **When**：在聊天输入框中输入 `/superpowers:brainstorming 为聊天界面添加快捷键支持` → 按 Enter 发送 → 等待 Agent 回复（可能需要等待 30-60 秒）
- **Then**：AskUserQuestion 面板出现 → 面板包含 header 区域（紫色圆形图标 + header 文字）→ 面板包含问题文本 → 面板包含选项列表 → 面板包含"自定义回答"输入区域

### 目标 2-4：键盘导航与聚焦高亮

**测试用例 2.1：↓ 方向键导航与聚焦高亮**
- **测试数据**：（依赖测试用例 1.1 的 AskUserQuestion 面板）
- **Given**：AskUserQuestion 面板已显示 → 点击面板容器区域使其获得焦点
- **When**：按一次 ↓ 键
- **Then**：第一个选项显示紫色聚焦高亮（`border-harness-accent` 边框 + `shadow-[0_0_0_2px_rgba(139,92,246,0.25)]` 外发光）→ 第一个选项背景色变为 `bg-[#1e1e3a]`

**测试用例 2.2：连续 ↓ 导航切换焦点**
- **测试数据**：（依赖测试用例 2.1 的聚焦状态）
- **Given**：第一个选项处于聚焦状态
- **When**：再按一次 ↓ 键
- **Then**：焦点移到第二个选项 → 第二个选项显示紫色聚焦高亮 → 第一个选项的高亮消失，恢复普通态

**测试用例 2.3：↑ 方向键向上导航**
- **测试数据**：（依赖测试用例 2.2 的聚焦状态）
- **Given**：第二个选项处于聚焦状态
- **When**：按一次 ↑ 键
- **Then**：焦点回到第一个选项 → 第一个选项重新显示聚焦高亮 → 第二个选项高亮消失

**测试用例 3.1：Ctrl+N 和 Ctrl+P 导航**
- **测试数据**：（依赖当前聚焦状态）
- **Given**：第一个选项处于聚焦状态
- **When**：按 Ctrl+N → 等待 → 按 Ctrl+P
- **Then**：Ctrl+N 使焦点向下移动 → Ctrl+P 使焦点向上移动 → 视觉高亮跟随焦点移动

**测试用例 4.1：Escape 键重置焦点**
- **测试数据**：（依赖当前聚焦状态）
- **Given**：某个选项处于聚焦状态
- **When**：按 Escape 键
- **Then**：所有选项的聚焦高亮消失 → 所有选项恢复普通态（`border-[#2a2a4e] bg-[#1a1a2e]`）

### 目标 5：鼠标悬停反馈

**测试用例 5.1：鼠标悬停显示柔和高亮**
- **测试数据**：（Escape 后无键盘聚焦状态）
- **Given**：Escape 后所有选项无聚焦高亮
- **When**：将鼠标悬停到第一个选项上
- **Then**：第一个选项显示柔和悬停高亮（`border-[#3a3a5c] bg-[#1c1c32]`，比聚焦态更柔和）

**测试用例 5.2：悬停不覆盖键盘聚焦态**
- **测试数据**：（需要键盘聚焦 + 鼠标悬停同时作用）
- **Given**：按 ↓ 使第一个选项处于键盘聚焦状态 → 显示紫色高亮
- **When**：将鼠标悬停到第二个选项上
- **Then**：第一个选项保持键盘聚焦的紫色高亮（`border-harness-accent` + glow）→ 第二个选项不显示任何高亮（键盘聚焦优先于悬停）

### 目标 6-7：单选提交交互

**测试用例 6.1：Enter 键提交聚焦选项**
- **测试数据**：（需要重新触发 AskUserQuestion）
- **Given**：AskUserQuestion 面板显示 → 按 ↓ 聚焦到第一个选项
- **When**：按 Enter 键
- **Then**：AskUserQuestion 面板消失 → Agent 继续处理（ThinkingBar 动画出现）→ 回答被提交给 Agent

**测试用例 7.1：点击选项先高亮后提交**
- **测试数据**：（需要 AskUserQuestion 面板）
- **Given**：AskUserQuestion 面板显示（无键盘聚焦）
- **When**：点击第一个选项
- **Then**：点击瞬间第一个选项显示聚焦高亮 → 约 300ms 后面板消失 → 回答被提交

### 目标 8-9：自定义回答

**测试用例 8.1：导航到自定义回答项，输入框自动聚焦**
- **测试数据**：（需要 AskUserQuestion 面板，选项数 N 个）
- **Given**：AskUserQuestion 面板显示 → 按 ↓ 聚焦到任意选项
- **When**：连续按 ↓ 直到焦点移到自定义回答项（即第 N+1 项，"自定义回答"分隔线区域）
- **Then**：自定义回答区域的分隔线变为紫色（`border-harness-accent`）→ 输入框自动获得焦点（光标在输入框内）

**测试用例 9.1：自定义回答输入和提交**
- **测试数据**：自定义回答 "这是一个自定义的测试回答"
- **Given**：自定义回答输入框已获得焦点
- **When**：输入 "这是一个自定义的测试回答" → 按 Enter
- **Then**：AskUserQuestion 面板消失 → Agent 收到自定义回答 → ThinkingBar 动画出现

### 目标 10：多选模式键盘操作

**测试用例 10.1：Space 键切换多选勾选**
- **测试数据**：（需要多选模式的 AskUserQuestion 面板，multiSelect: true）
- **Given**：多选模式的 AskUserQuestion 面板显示 → 按 ↓ 聚焦到第一个选项
- **When**：按 Space 键
- **Then**：第一个选项的勾选状态切换（☐ → ☑）→ 选项边框和背景显示选中态 → 再按 Space 勾选取消（☑ → ☐）

**测试用例 10.2：多选确认按钮提交**
- **测试数据**：（依赖测试用例 10.1，已勾选至少一个选项）
- **Given**：已用 Space 勾选 1-2 个选项
- **When**：点击"确认"按钮
- **Then**：面板消失 → Agent 收到所有勾选项的答案 → ThinkingBar 动画出现

> **注意**：多选模式取决于 brainstorming skill 是否产生 multiSelect: true 的问题。如果第一个问答周期未出现多选问题，后续问答中可能出现。若整个测试周期均未触发多选，将记录为"无法验证"。

## 验证方案

### 通过标准
所有测试用例的 Then 步骤均验证通过。其中：
- **视觉验证**：通过截图对比确认聚焦高亮、悬停高亮、选中态的视觉差异
- **功能验证**：通过面板消失、ThinkingBar 出现确认提交成功
- **多选模式**：如无法触发则标记为"未覆盖"，不影响其他测试结果的通过判定

### 失败标准
- 键盘导航无响应（按 ↓ 后焦点未移动）
- 视觉状态异常（聚焦/悬停/选中态无法区分）
- 提交失败（选择后面板未消失或 Agent 未继续处理）

## 操作规则

### 默认规则
1. 直接用浏览器进行端到端测试，不写任何脚本
2. 以本文档的测试步骤为操作流程依据
3. 输入框操作：输入完成后用回车键或点击"发送"/"提交"按钮，**禁止**用 `\n` 字符
4. 下拉选择：用 select 操作，不要手动模拟键盘
5. 按钮点击：直接用 click 操作定位按钮元素
6. 等待策略：操作后等待页面响应（await_element / await_text），不要假设即时生效

### 项目特定规则
1. Agent 回复较慢（LLM 处理），使用 await_text 或截图确认状态，等待时间可能需要 30-60 秒
2. AskUserQuestion 触发时机不确定，需耐心等待
3. 键盘操作前需要先点击 AskUserQuestion 面板容器使其获得焦点（tabIndex=0）
4. 视觉状态验证通过截图 + 读取截图文件分析
5. brainstorming skill 会产生多轮问答，可利用后续问题继续测试
